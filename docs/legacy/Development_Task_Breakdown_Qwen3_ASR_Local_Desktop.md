# 开发任务清单：本地 ASR 语音转文字桌面应用

## 1. 文档信息

- 文档名称：开发任务清单（Development Task Breakdown）
- 对应 PRD：`PRD_Qwen3_ASR_Local_Desktop.md`
- 对应技术设计：`Technical_Design_Qwen3_ASR_Local_Desktop.md`
- 文档版本：v1.0
- 编写日期：2026-04-17

## 2. 使用说明

本清单用于把项目从“文档阶段”推进到“可开发、可联调、可验收”的执行阶段。

任务拆分原则：

1. 先验证核心链路，再做桌面包装。
2. 先保稳定性，再做体验优化。
3. 所有任务默认围绕 Windows 原生 `transformers + CUDA` 实现。

## 3. 里程碑总览

### M0：工程初始化

目标：

1. 建立基础仓库结构。
2. 建立 Rust GUI 与 Python Worker 的独立开发环境。
3. 建立最小运行链路和文档基线。

### M1：后端核心链路验证

目标：

1. 跑通 `pyannote -> segment slice -> Qwen3-ASR -> transcript merge`。
2. 能导出基础 Markdown。
3. 能在命令行模式下稳定执行完整任务。

### M2：桌面 MVP 联调

目标：

1. Rust GUI 可创建任务并启动 Worker。
2. GUI 可显示进度、结果和导出状态。
3. MVP 达到 PRD 验收口径。

### M3：体验增强

目标：

1. 提升 transcript 可读性。
2. 增加发言人重命名、任务历史和诊断能力。
3. 完成日志和可观测性补齐。

### M4：工程化发布

目标：

1. 完成打包与安装流程。
2. 补足测试、错误恢复与环境诊断。
3. 准备首个可分发版本。

## 5. 依赖关系概览

关键依赖顺序如下：

1. `M0` 完成后才能进入任何实现任务。
2. `T103-T110` 决定后端核心链路是否成立，是全项目关键路径。
3. `T203-T207` 决定桌面端是否能形成真正可用的 MVP。
4. `M3` 和 `M4` 都建立在 `T207` 完成后。

## 6. 建议优先级

当前建议的真实执行顺序：

1. `T001-T003`
2. `T101-T106`
3. `T103-T110`
4. `T111`
5. `T201-T207`
6. `T301-T304`
7. `T401-T404`

## 7. 风险追踪任务

建议单独跟踪以下风险项：

1. `R001`：Windows 下 `PyTorch/CUDA` 兼容性验证
2. `R002`：pyannote 模型获取与授权流程验证
3. `R003`：Qwen3-ASR 长音频片段转写稳定性验证
4. `R004`：打包后 Python Worker 运行时分发方案验证

每个风险项都应输出一份短结论：

1. 是否成立
2. 影响范围
3. 规避方案

## 8. MVP 完成定义

满足以下条件即可认定 MVP 完成：

1. 用户可从 GUI 选择本地音频文件。
2. 用户可输入背景文本、术语表与输出目录。
3. 系统可在 Windows 原生 `transformers + CUDA` 路径上运行。
4. 系统可通过 `pyannote + Qwen3-ASR` 链路生成 transcript。
5. 输出结果为带 `Speaker` 标签和段级时间戳的 Markdown。
6. GUI 可预览结果并导出 `.md` 文件。

## 9. 当前建议

如果下一步开始进入编码阶段，建议先做三件事：

1. 创建 Rust 和 Python 两个项目骨架。
2. 优先做命令行版后端全链路验证，不要一上来先做复杂 GUI。
3. 等 `T111` 跑通后，再推进 GUI 联调和视觉层实现。
