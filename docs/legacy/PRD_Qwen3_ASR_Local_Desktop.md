# PRD：本地 ASR 语音转文字桌面应用（基于 Qwen3-ASR-1.7B）

## 1. 文档信息

- 产品名称：本地 ASR 语音转文字桌面应用
- PRD 版本：v1.1
- 编写日期：2026-04-16
- 目标平台：Windows 桌面优先
- 部署方式：本地离线优先，模型下载完成后支持纯本地运行

## 2. 项目背景

用户希望构建一款本地 ASR 桌面软件，核心需求如下：

1. 使用 `Qwen/Qwen3-ASR-1.7B` 作为核心语音转文字模型。
2. 充分利用本地 `RTX 5060 Ti 16GB` GPU 做推理。
3. 提供本地 GUI，前端采用 Rust 技术栈。
4. 支持本地录音文件导入、语音识别、带时间戳与说话人标签的 Markdown 输出与导出。
5. 在识别前允许用户输入背景文本、术语、人名、专有名词等上下文，让系统尽量输出更贴合业务语境的识别结果。
6. 采用固定的组合式技术方案实现“发言人识别 + 语音识别 + 时间戳”。

本次版本对技术路线做出明确收敛：

1. `pyannote.audio` 负责发言人分离与时间边界切分。
2. `Qwen3-ASR-1.7B` 负责对每个切分片段逐段转写。
3. 最终按全局时间轴合并为带 `Speaker` 标签的 transcript。
4. 推理后端固定采用 Windows 下的 `transformers + CUDA` 路线。
5. 当前版本不考虑 `WSL` 或 `vLLM` 方案。

## 3. 产品愿景

打造一款面向个人与小团队的、本地优先、隐私友好、可利用消费级 GPU 加速的语音转文字桌面工具。用户可以直接选择本机音频文件，填写背景信息后开始识别，并得到结构化、可读性强、可保存为 `.md` 文件的带说话人标签转写稿。

## 4. 产品目标

### 4.1 核心目标

1. 用户可以从本地选择音频文件并发起识别任务。
2. 识别过程默认调用本地 GPU，而不是退回 CPU。
3. 输出结果为 Markdown，并带段级时间戳与 `Speaker` 标签。
4. 支持通过背景文本、术语表、专有名词表来提升识别结果的业务适配度。
5. 支持结果预览与导出保存。
6. 在 MVP 阶段即交付“发言人切分 + 逐段转写 + 时间轴合并”的完整主流程。

### 4.2 非目标

1. MVP 不做云端账号、同步、团队协作。
2. MVP 不做实时会议录音和在线流式麦克风转写。
3. MVP 不做移动端。
4. MVP 不做真正的人名级说话人身份识别，只输出 `Speaker 1/2/3...` 这类匿名标签。
5. MVP 不做复杂文档编辑器，只提供基础预览和导出。
6. 当前版本不考虑 `WSL`、`vLLM` 或多后端切换。

## 5. 目标用户

### 5.1 主要用户

1. 采访、会议、课程、访谈录音的整理人员。
2. 需要本地离线转写、注重隐私的数据工作者。
3. 经常处理专业术语、人名、公司名、产品名的知识工作者。

### 5.2 典型使用场景

1. 会议录音转纪要。
2. 访谈录音转文字初稿。
3. 课程音频转讲义素材。
4. 多人讨论音频的发言人整理与留档。

## 6. 产品定位

本产品定位为“本地桌面转写工具”，而非“云端 SaaS 录音服务”。因此交互逻辑必须围绕“本地文件”“本地缓存”“本地导出”“本地模型推理”展开。

## 7. 核心价值主张

1. 本地运行：敏感音频不离开本机。
2. GPU 加速：利用消费级显卡缩短等待时间。
3. 可读结果：输出结构清晰的 Markdown，而不只是纯文本。
4. 说话人可区分：默认输出按时间轴排序的 `Speaker` 标签稿。
5. 语境增强：支持用户输入背景信息和术语表，提升专有名词命中率。

## 8. 关键假设与边界

### 8.1 官方能力边界

1. `Qwen3-ASR-1.7B` 官方支持语言识别和语音识别。
2. 官方支持长音频转写。
3. 官方提供基于 `transformers` 的本地推理路径。
4. 官方文档未明确将“speaker diarization / speaker labels”列为模型原生输出能力。
5. 因此发言人识别必须通过外部 diarization 模块补齐。

### 8.2 硬件与系统边界

1. 用户当前为 `RTX 5060 Ti 16GB`。
2. 该卡显存容量对 `Qwen3-ASR-1.7B + pyannote` 单机部署具有可行性。
3. 但用户当前环境是 Windows，本项目必须考虑 Windows 下 GPU 推理的工程可落地性。
4. 本项目仅考虑 Windows 原生环境下的 `transformers + CUDA` 方案。

## 9. 推荐技术方案

## 9.1 总体架构结论

推荐采用“双层架构”：

1. Rust 原生桌面 GUI 作为主应用。
2. Python 本地推理服务作为语音引擎。

其中：

1. GUI 层负责文件选择、任务配置、进度展示、结果预览、导出。
2. 推理层负责音频预处理、发言人切分、逐段 ASR、结果合并、术语归一化与 Markdown 输出。

这样设计的原因是：

1. Qwen3-ASR 官方主要提供的是 Python 推理生态。
2. Rust 直接承接模型推理的集成成本高、资料少、维护复杂。
3. Rust 做 GUI、Python 做模型推理，是当前最稳妥的工程拆分。

## 9.2 GUI 技术栈建议

推荐技术栈：`Rust + Slint`

推荐理由：

1. 满足“前端用 Rust 来写”的要求，不依赖 Web 技术栈。
2. 适合本地原生桌面应用。
3. 对文件选择、列表、进度、表单、预览面板等常见桌面交互足够友好。
4. 适合后续打包为 Windows 桌面应用。

备选方案：`egui`

适用条件：

1. 若项目优先追求快速做出内部工具原型。
2. 若可以接受 UI 风格更偏工程工具感。

结论：

1. 若目标是“长期使用的桌面产品”，推荐 `Slint`。
2. 若目标是“最短时间做 MVP 原型”，可先用 `egui`，后续再演进。

## 9.3 推理技术栈建议

推荐技术栈：

1. Python 3.11
2. `torch`
3. `transformers`
4. `qwen-asr`
5. `pyannote.audio`
6. `ffmpeg`
7. `pydub` 或等价音频切片工具

模型组合：

1. ASR 主模型：`Qwen/Qwen3-ASR-1.7B`
2. 发言人分离模型：`pyannote.audio` 对应 speaker diarization pipeline

## 9.4 唯一推理后端方案

本项目固定采用：

`Windows 原生 transformers + CUDA`

选择理由：

1. Qwen 官方示例明确支持 `device_map="cuda:0"`。
2. 该方案更符合本地桌面产品的安装和维护习惯。
3. 单用户桌面场景对极限吞吐要求不高，更强调工程稳定性与可落地性。
4. 用户已明确要求不考虑 `WSL` 路线，因此不引入额外系统复杂度。

结论：

1. 当前版本不设计多后端切换。
2. 所有性能优化均基于 Windows 原生 `transformers` 方案展开。

## 9.5 核心组合式识别方案

本产品采用唯一主链路：

1. 使用 `pyannote` 对整段音频做 speaker diarization。
2. 得到一组带开始时间、结束时间、speaker 标签的时间片段。
3. 按时间片段从原音频中切出子音频。
4. 使用 `Qwen3-ASR-1.7B` 对每个子音频逐段转写。
5. 以 `pyannote` 给出的时间边界作为该段 transcript 的时间戳。
6. 按全局时间顺序合并为最终 transcript。

### 方案说明

该方案中：

1. 发言人标签来源于 `pyannote`。
2. 文本内容来源于 `Qwen3-ASR-1.7B`。
3. 段级时间戳来源于 `pyannote` 切分边界。

### 产出格式

每条 transcript 记录至少包含：

1. `speaker`
2. `start_ms`
3. `end_ms`
4. `text`

### 选择该方案的原因

1. 路径单一，工程可控。
2. 可直接满足“发言人 + 转写 + 时间戳”的组合需求。
3. 不依赖 `Qwen3-ASR` 的原生 speaker 能力假设。
4. 更适合多人会议、访谈类录音。

## 9.6 RTX 5060 Ti 16GB 适配策略

### 适配判断

工程上可推断：

1. 在单卡 `16 GB VRAM` 下运行 `Qwen3-ASR-1.7B` 与 `pyannote` 具备现实可行性。
2. 但仍需要限制并发、控制切片长度，并优先保证稳定性。

### 默认运行策略

1. 默认同一时间只跑 1 个识别任务。
2. 默认关闭多任务并发。
3. 发言人切分与 ASR 不同时并行执行，优先采用串行阶段式处理。
4. 逐段转写时采用保守批大小，避免 OOM。
5. 对过长片段进行二次切分，避免单段音频过长导致显存峰值抖动。

### 精度建议

1. 优先尝试 `bfloat16`。
2. 若 Windows / 驱动 / PyTorch 对 Blackwell 兼容性存在问题，则回退到 `float16`。
3. 应在设置中提供“自动 / bf16 / fp16”选项，但默认自动。

## 11. 组合式发言人识别方案设计

## 11.1 产品定义

当前版本中的“识别说话人”定义为：

1. 将多人音频切分为多个发言片段。
2. 给每个片段分配 `Speaker 1/2/3...` 标签。
3. 不承诺直接识别出真实姓名。

## 11.2 主流程

主流程如下：

1. 对原始音频做标准化解码。
2. 使用 `pyannote` 做 diarization，产出 `(speaker, start, end)` 片段列表。
3. 对每个片段裁切出对应的音频片段。
4. 对每个片段调用 `Qwen3-ASR-1.7B` 转写。
5. 生成结构化记录：

```text
{
  speaker: "Speaker 1",
  start_ms: 0,
  end_ms: 8000,
  text: "大家晚上好，我们开始今天的项目周会。"
}
```

6. 按 `start_ms` 升序排列。
7. 合并为最终 transcript 并输出为 Markdown。

## 11.3 合并策略

1. 以 `pyannote` 片段边界作为时间戳权威来源。
2. 若相邻片段属于同一 speaker 且间隔极短，可做可选合并，提升可读性。
3. 若某片段过短且转写为空，可标记为空白片段并在最终输出时省略。
4. 若同一 speaker 连续多段短句，可以按配置选择是否合并为一个更长段落。

## 11.4 结果可读性策略

为避免 transcript 过碎，建议引入以下规则：

1. diarization 结果先进行最小片段时长过滤。
2. 小于阈值的极短片段可并入前后同 speaker 片段。
3. 同一 speaker 的连续短片段可在时间连续条件下合并。

## 11.5 后续增强方向

1. 支持用户手动重命名 `Speaker 1/2/3`。
2. 支持对 transcript 做句级再切分。
3. 支持在预览页手动编辑文本并再次导出。

## 12. 长音频处理策略

### 设计原则

1. 长音频先整体做 speaker diarization。
2. 再按 speaker 片段逐段转写。
3. 若某一 speaker 片段过长，则在保留 speaker 标签的前提下进一步二次切分。

### 推荐流程

1. 将输入音频统一解码为标准格式。
2. 运行 `pyannote` 生成时间轴片段。
3. 对超过阈值的片段进行二次切分。
4. 对每个最终片段独立转写。
5. 最终按原始时间轴合并并回填全局起止时间。

### 好处

1. 降低 OOM 风险。
2. 更适配多人会议或访谈录音。
3. 有利于失败重试。

## 13. 非功能需求

## 13.1 性能目标

以下为产品目标值，不代表当前已实测数据：

1. 首次模型下载后，常规单任务识别应默认走 GPU。
2. 60 分钟音频应在可接受时间内完成带 `Speaker` 标签的 Markdown 输出。
3. 默认配置下显存占用应尽量控制在单卡可承受范围内。
4. 采用 diarization + 逐段转写链路时允许处理时间上升，但不能导致主流程不可用。

## 13.2 稳定性目标

1. 任务失败时必须保留错误日志。
2. 任务中断时不得损坏已生成结果。
3. 导出失败时必须给出清晰原因。

## 13.3 可用性目标

1. 非技术用户可在 10 分钟内完成首次转写。
2. 默认配置应尽量少，让用户只需“选文件 + 填背景 + 开始识别”。

## 13.4 隐私目标

1. 所有识别流程默认在本地执行。
2. 音频文件不上传到远程服务。
3. 导出结果默认保存在用户本地目录。

## 14. 系统架构草案

```text
Rust Desktop App (Slint)
    |
    |-- File Picker / Task Config / Preview / Export
    |
    |-- Local IPC or HTTP (127.0.0.1)
            |
            v
      Python Inference Worker
            |
            |-- ffmpeg decode / resample
            |-- pyannote diarization
            |-- segment slicing
            |-- Qwen3-ASR-1.7B transcription
            |-- timeline merge
            |-- terminology normalization
            |-- markdown generation
```

## 15. 数据流设计

1. 用户选择本地文件。
2. GUI 将任务配置写入本地任务对象。
3. GUI 调用本地推理服务。
4. 推理服务完成音频预处理。
5. 调用 `pyannote` 得到 speaker 时间片段。
6. 按片段切出音频子段。
7. 调用 `Qwen3-ASR-1.7B` 逐段得到文本。
8. 将 `speaker + 时间边界 + 文本` 合并为结构化 transcript。
9. 对识别结果执行术语归一化。
10. 生成 Markdown 文本。
11. 将结果保存为 `.md` 并回传 GUI 展示。

## 18. 风险与应对

### 风险 1：Windows 下 GPU 推理兼容性

风险说明：

1. RTX 50 系为较新的 Blackwell 架构。
2. 不同 `PyTorch / CUDA` 组合在 Windows 上可能存在兼容差异。

应对：

1. MVP 先做一轮环境矩阵验证。
2. 固定推荐驱动、CUDA 与 PyTorch 版本组合。

### 风险 2：diarization 切分质量影响最终 transcript

风险说明：

1. 若 `pyannote` 切分过碎、串 speaker 或漏切，会直接影响输出可读性。

应对：

1. 增加最小片段长度与相邻同 speaker 合并规则。
2. 在预览页保留后续人工修订入口。

### 风险 3：长音频显存波动

风险说明：

1. 长音频在切分、裁切和逐段转写过程中可能出现显存峰值抖动。

应对：

1. 强制分段处理。
2. 限制并发。
3. 默认采用保守推理参数。

### 风险 4：上下文增强效果不稳定

风险说明：

1. 仅依赖提示词未必总能稳定命中术语。

应对：

1. 加入术语归一化后处理。
2. 在预览页允许手动校对和再次导出。

## 19. 最终产品决策摘要

1. 模型选择：`Qwen/Qwen3-ASR-1.7B`
2. 发言人方案：`pyannote.audio`
3. 时间戳方案：使用 `pyannote` 片段边界作为段级时间戳
4. GUI 方案：`Rust + Slint`
5. 推理后端：`Python + transformers + CUDA`，仅限 Windows 原生环境
6. 核心流水线：`pyannote 切 speaker -> Qwen3-ASR 逐段转写 -> 按时间轴合并 transcript`
7. 输出格式：结构化 Markdown，本地导出为 `.md`
