# PRD：本地 ASR 桌面应用 Tauri + Vue + TypeScript 稳定重构

## 1. 文档信息

- 产品名称：本地 ASR 听记助手
- 文档类型：产品重构 PRD
- PRD 版本：v1.0
- 编写日期：2026-06-07
- 目标平台：Windows 桌面优先
- 推荐技术方案：Tauri + Vue 3 + TypeScript + 现有 Python Worker
- 重构原则：稳定优先、前端重构、业务逻辑最小变更

## 2. 项目背景

当前产品已经形成可运行的本地桌面应用原型：

1. `apps/desktop-rust` 使用 Rust + Slint 承载桌面 GUI。
2. `apps/worker-python` 使用 Python 承载 ASR、speaker diarization、Markdown 导出等核心业务逻辑。
3. 桌面端通过 JSON Lines over stdio 管理 Python worker。
4. 当前 GUI 已包含任务配置、进度展示、Markdown 预览/编辑、大模型总结、输出导出、日志等能力。

随着产品继续演进，现有 Rust + Slint GUI 面临以下问题：

1. 复杂文档编辑、Markdown 渲染、代码高亮、搜索、虚拟滚动等能力在 Web 前端生态中更成熟。
2. GUI 状态和交互继续用 Slint 扩展，前端开发效率和组件生态会受限。
3. 当前业务核心在 Python worker，Rust GUI 层过厚会增加后续维护成本。
4. 产品需要保留独立桌面应用形态，同时提升前端体验和工程迭代速度。

因此，本次重构选择：

```text
Tauri 桌面壳 + Vue 3 / TypeScript 前端 + 现有 Python Worker
```

其中 Rust 仅作为 Tauri 后端薄壳，负责桌面能力和 worker 管理；Vue + TypeScript 负责完整产品前端；Python worker 继续承载稳定业务逻辑。

## 3. 产品愿景

将当前原型演进为一款稳定、轻量、体验更完整的本地听记桌面产品。用户可以在独立桌面应用中完成录音导入、转写配置、转写进度跟踪、Markdown 编辑预览、大模型总结和结果导出，同时保留当前本地优先、隐私友好、GPU 加速的核心价值。

## 4. 重构目标

### 4.1 核心目标

1. 保持当前 Python ASR 业务链路稳定，不在第一阶段重写推理逻辑。
2. 使用 Tauri 替换现有 Slint 桌面 GUI 壳。
3. 使用 Vue 3 + TypeScript 重建前端界面、状态管理和 Markdown 工作区。
4. 保持现有 JSON Lines worker 通信模式，降低联调风险。
5. 提供高性能、轻量的 Markdown 编辑与渲染体验。
6. 保持旧版 `desktop-rust` 可回退，直到新版功能达到验收标准。

### 4.2 体验目标

1. 用户能在新版桌面应用中完成与当前版本等价的主流程。
2. 长 Markdown 文档打开、编辑、预览时界面保持可交互。
3. 任务进度、worker 状态、错误信息和日志入口更清晰。
4. 设置、模板、总结配置等长期配置更容易管理。
5. 产品界面从工程原型演进为稳定桌面工具体验。

### 4.3 工程目标

1. 前端、Tauri shell、Python worker 职责边界清晰。
2. worker 协议有显式 schema 和兼容策略。
3. 新旧桌面端可在一段时间内并行存在。
4. 构建、开发启动、打包流程可重复。
5. 支持后续逐步替换或增强 worker 通信机制，但不作为本期目标。

## 5. 非目标

本次重构不覆盖以下内容：

1. 不重写 `pyannote + Qwen3-ASR` 推理链路。
2. 不引入云端账号、同步、多用户协作。
3. 不做实时麦克风流式转写。
4. 不做跨平台发布承诺，Windows 优先。
5. 不在第一阶段引入本地 HTTP 服务作为默认通信模式。
6. 不做完整所见即所得 Markdown 编辑器。
7. 不追求一次性删除旧 Rust/Slint 应用。
8. 不改变当前模型目录、输出目录、配置文件的基本约定。

## 6. 目标用户与核心场景

### 6.1 目标用户

1. 会议、访谈、电话录音的整理人员。
2. 需要本地离线处理敏感音频的知识工作者。
3. 需要将长音频转为结构化 Markdown 文稿的用户。
4. 经常基于转写稿生成总结、纪要、访谈记录和分析材料的用户。

### 6.2 核心场景

1. 用户导入本地录音文件，填写背景信息和术语表后开始转写。
2. 用户观察转写进度、worker lane 状态、当前阶段和错误提示。
3. 用户在任务完成后打开 Markdown 转写稿，进行检查、编辑和保存。
4. 用户基于转写稿选择总结模板，生成结构化 Markdown 总结。
5. 用户导出转写稿、总结稿或最终稿。
6. 用户查看运行日志，排查 Python、CUDA、模型路径和任务失败问题。

## 8. 信息架构

新版桌面应用建议采用以下一级导航：

1. `工作台`：发起转写、查看 worker lane、查看当前任务。
2. `Markdown`：打开、编辑、预览、保存转写稿。
3. `总结`：选择输入稿、配置模板、生成总结、导出总结。
4. `历史`：查看本地任务记录和输出文件。
5. `设置`：模型路径、Python 环境、API 配置、输出目录、日志。

### 8.1 工作台

工作台是默认首页，承载主转写流程：

1. 音频文件选择。
2. 输出配置。
3. 转写参数。
4. 背景上下文。
5. 术语与替换规则。
6. 开始转写。
7. 双 worker lane 状态卡片。
8. 当前任务进度和最近错误。

### 8.2 Markdown 工作区

Markdown 工作区承载长文本处理：

1. 左侧源码编辑器。
2. 右侧渲染预览。
3. 大文档性能模式。
4. 搜索与定位。
5. 保存、另存为、导出。
6. 当前文件脏状态提示。

### 8.3 总结

总结页承载基于转写稿的大模型生成：

1. 选择当前 Markdown 输入。
2. 选择总结 profile。
3. 编辑 Prompt。
4. 生成总结。
5. 编辑总结结果。
6. 导出总结 Markdown。

### 8.4 历史

历史页承载本地任务索引：

1. 按时间查看任务。
2. 查看源文件、输出文件、任务状态。
3. 快速打开转写 Markdown。
4. 快速打开输出目录。
5. 查看任务 JSON 和日志。

### 8.5 设置

设置页承载长期配置：

1. Python 运行时检测。
2. CUDA 和 GPU 诊断。
3. 模型路径检测。
4. 输出目录默认值。
5. 总结 API 配置。
6. Prompt 模板管理。
7. 日志路径和清理策略。

## 9. 核心用户流程

### 9.1 转写流程

1. 用户进入工作台。
2. 用户选择本地音频文件。
3. 用户确认输出目录和输出文件名。
4. 用户填写背景文本、术语和替换规则。
5. 用户点击开始转写。
6. 前端通过 Tauri command 提交 `RunJobRequest`。
7. Tauri 后端将请求转发给 Python worker。
8. Python worker 持续返回进度事件。
9. 前端实时更新 worker lane 和任务进度。
10. 任务完成后前端加载 Markdown 输出。
11. 用户进入 Markdown 工作区检查和编辑。

### 9.2 Markdown 编辑流程

1. 用户打开转写结果或手动选择 Markdown 文件。
2. 编辑器加载 Markdown 源文。
3. 预览面板按节流策略增量更新。
4. 用户保存或另存为。
5. 保存成功后更新文件状态和历史记录。

### 9.3 总结流程

1. 用户选择当前转写稿作为总结输入。
2. 用户选择总结模板或自定义 Prompt。
3. 用户点击生成总结。
4. 前端通过 Tauri command 调用总结服务。
5. 总结结果回填到编辑区。
6. 用户编辑后导出 Markdown。

### 9.4 错误恢复流程

1. worker 启动失败时，前端展示用户可理解的原因。
2. 任务失败时，前端展示阶段、摘要、日志入口。
3. Python worker 异常退出时，Tauri shell 标记 worker 不可用并允许重启。
4. 保存失败时，前端展示路径和权限相关提示。
5. 用户可以重新提交任务，旧任务输出不被覆盖，除非用户确认。

## 11. 非功能需求

### 11.1 稳定性

1. GUI 崩溃不应破坏已完成的输出文件。
2. Python worker 异常退出时，前端必须进入可恢复状态。
3. 任务取消、暂停、恢复必须对当前 job_id 有明确绑定。
4. 新版应用不得改变旧输出目录中已生成文件的格式兼容性。

### 11.2 性能

1. 应用冷启动目标：开发机环境下 3 秒内展示主窗口。
2. Markdown 文件小于 2 MB 时，打开和预览应保持流畅。
3. Markdown 文件大于 2 MB 时，进入性能模式，优先保证编辑可用。
4. 进度事件刷新应合并节流，避免高频事件造成前端重渲染。
5. 前端 bundle 应保持轻量，避免引入完整 IDE 级依赖，除非有明确收益。

### 11.3 安全与隐私

1. 默认本地处理音频，源音频不上传。
2. 只有用户主动使用总结功能时，Markdown 内容才会发送到配置的 API。
3. 前端渲染 Markdown 时不得执行任意脚本。
4. 日志中避免记录完整 API key。
5. 错误报告默认仅落地本地日志。

### 11.4 可维护性

1. TypeScript 类型必须覆盖 worker request、event、result。
2. 前端状态管理必须集中，避免页面之间复制任务状态。
3. Tauri command 必须保持薄层，不承载 ASR 业务逻辑。
4. Python worker 业务逻辑变更必须有单独验收，不与前端迁移耦合。

## 12. Markdown 技术要求

### 12.1 推荐组件

1. 编辑器：CodeMirror 6。
2. 渲染器：`markdown-it` 或 `micromark` 系列。
3. 安全净化：DOMPurify。
4. 代码高亮：按需加载的 lightweight highlighter。
5. 大文档优化：分块渲染、节流更新、必要时虚拟滚动。

### 12.2 设计原则

1. Markdown 源码编辑优先于所见即所得。
2. 预览可以延迟更新，但编辑输入不能卡顿。
3. 对长文档只渲染必要区域或降低预览刷新频率。
4. 表格、代码块、标题目录是高优先级支持对象。
5. HTML 渲染默认受限，避免安全风险。

### 12.3 性能验收样本

至少使用以下样本验证：

1. 10 KB 短会议纪要。
2. 500 KB 常规访谈稿。
3. 2 MB 长转写稿。
4. 5 MB 极限长文稿。

验收口径：

1. 10 KB 和 500 KB 文档应即时打开并流畅编辑。
2. 2 MB 文档可编辑，预览允许节流。
3. 5 MB 文档必须可打开，不要求完整实时预览。

## 13. 业务逻辑稳定策略

### 13.1 Worker contract 先行

重构前必须先冻结一版 worker contract：

1. `RunJobRequest`
2. `JobProgress`
3. `JobResult`
4. `WorkerEnvelope`
5. `SummaryRequest`
6. `SummaryResult`
7. 错误结构

所有新前端和 Tauri command 都围绕 contract 开发。

### 13.2 Python worker 最小改动

第一阶段只允许以下 Python worker 改动：

1. 补齐协议字段。
2. 修复明确 bug。
3. 增加健康检查命令。
4. 增加更清晰的错误字段。
5. 增加必要的测试夹具。

不允许在第一阶段重构 ASR pipeline、模型加载、diarization 算法和导出格式。

### 13.3 新旧应用并行

在新版达到功能等价前：

1. 保留 `apps/desktop-rust`。
2. 新增 `apps/desktop-tauri`。
3. 两个桌面端共享 `apps/worker-python`。
4. 两个桌面端输出相同 job 结构和 Markdown 格式。
5. 同一测试音频在两个桌面端下结果路径和主要字段应一致。

## 15. 技术边界

### 15.1 Vue + TypeScript 前端职责

1. 页面渲染。
2. 表单校验。
3. 前端状态管理。
4. Markdown 编辑预览。
5. 用户操作入口。
6. 任务状态展示。
7. 错误信息展示。

### 15.2 Tauri 后端职责

1. 创建桌面窗口。
2. 文件选择和本地文件读写。
3. 启动和管理 Python worker。
4. 转发 worker request。
5. 接收 worker event 并推送到前端。
6. 管理日志路径。
7. 暴露配置读写 command。

Tauri 后端不负责：

1. ASR 推理。
2. speaker diarization。
3. Markdown 业务格式生成。
4. 总结 Prompt 业务决策。

### 15.3 Python worker 职责

1. 环境检测。
2. 模型加载。
3. 音频处理。
4. diarization。
5. ASR 转写。
6. transcript 合并。
7. Markdown 和 JSON 导出。
8. 任务级错误返回。

## 17. 验收标准

### 17.1 功能验收

1. 新版应用可完成一次完整转写任务。
2. 新版应用可打开、编辑、保存转写 Markdown。
3. 新版应用可生成和导出总结 Markdown。
4. 新版应用可展示 worker 进度、暂停、恢复、终止。
5. 新版应用可读取现有配置文件。
6. 新版应用可展示日志和诊断信息。

### 17.2 一致性验收

1. 同一输入音频在新旧应用下使用同一 worker 输出格式兼容。
2. `transcript.json` 字段不因前端重构而破坏。
3. Markdown 输出模板不因前端重构而改变。
4. 配置文件路径和含义保持兼容。

### 17.3 稳定性验收

1. worker 崩溃后 GUI 不崩溃。
2. 取消任务后可重新提交新任务。
3. Markdown 保存失败时不丢失编辑区内容。
4. API 调用失败时总结输入不丢失。
5. 旧版应用仍可继续启动和执行任务。

## 18. 风险与应对

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| Tauri worker 管理与现有 Rust worker_client 迁移不完整 | 任务无法稳定执行 | 先迁移协议和最小 run_job，再迁移暂停/恢复/终止 |
| Markdown 大文档渲染卡顿 | 核心体验下降 | 使用节流、分块渲染、性能模式和必要的虚拟滚动 |
| 前端状态复杂度上升 | bug 增多 | 使用集中 store，并将 worker event reducer 类型化 |
| Python worker 被误改 | 业务逻辑回归 | 第一阶段冻结 worker pipeline，只允许协议补强 |
| 打包体积和依赖复杂 | 发布困难 | Tauri shell、Python runtime、models 分层打包 |
| 新旧配置不兼容 | 用户迁移失败 | 保持配置文件路径和字段兼容，必要时提供迁移函数 |

## 19. 待决策问题

1. 前端状态管理使用 Pinia 还是轻量 composable store。
2. Markdown 渲染器选择 `markdown-it` 还是 `micromark`。
3. 是否保留当前双 worker lane，还是先降级为单 lane 后再恢复。
4. 历史任务索引使用文件扫描还是单独维护本地索引文件。
5. Windows 打包是否内置 Python runtime，还是继续依赖项目级虚拟环境。
6. 总结 API 调用保留在 Tauri 后端，还是迁移到 Python service 层。

## 20. 推荐结论

本次重构应采用：

```text
Tauri + Vue 3 + TypeScript + 现有 Python Worker
```

产品重构的优先级不是“重写更多代码”，而是：

1. 先冻结 worker contract。
2. 再替换 GUI。
3. 再优化 Markdown 工作区。
4. 最后完成打包和默认入口切换。

只要 Python worker 和输出格式保持稳定，新版前端可以在较低风险下逐步替代现有 Rust + Slint GUI，并为后续产品体验迭代提供更成熟的前端生态。
